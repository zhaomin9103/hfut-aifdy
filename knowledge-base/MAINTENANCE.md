# MAINTENANCE —— 知识库维护手册

> 怎样让知识库保持「新鲜、可信、可检索」。

---

## 1. 核心原则

1. **持久 vs 一次性分离**：稳定结论进 `knowledge-base/`，临时草稿留在 `workspace/`。
2. **就近回写**：开发中得到的稳定结论，立刻写入对应目录，不要攒着。
3. **留痕**：每次变更都要在 `log.md` 追加一行（append-only，不可删改历史）。
4. **格式合规**：遵守 `SCHEMA.md`，否则校验脚本会拒绝。

---

## 2. 各目录的更新时机

| 目录 | 什么时候更新 | 谁来更新 |
|------|--------------|----------|
| `system/` | 架构、路由、依赖、调用链发生变化时 | 改动者 |
| `modules/` | 新增/删除/重命名功能模块时 | 改动者 |
| `pitfalls/` | 踩到新坑、坑被解决时 | 踩坑者 / 解决者 |
| `conventions/` | 团队达成新约定、约定变更时 | 提议者 + 评审 |
| `insights/` | 同一模式重复出现 ≥ 2 次时 | 发现者 |
| `raw/` | 收到 PRD、纪要等原件时（只读归档） | 收件人 |

---

## 3. 标准操作流程（SOP）

### 新增一篇知识文档
1. 确认归属目录（参照 README 导览表）。
2. 复制 `SCHEMA.md` 的 Front Matter 模板，填好元数据。
3. 撰写正文，引用源码处带相对路径 + 行号。
4. 跑一遍 `SCHEMA.md` 第 5 节的校验清单。
5. 在 `log.md` 追加记录。

### 更新已有文档
1. 修改正文。
2. 把 Front Matter 的 `updated` 改为今日。
3. 若内容已不适用，把 `status` 改为 `deprecated`（**不要直接删**，保留可追溯）。
4. 在 `log.md` 追加记录。

### 废弃文档
- 优先标记 `status: deprecated` 并说明替代文档。
- 仅当确认无引用、无追溯价值时才物理删除，并在 `log.md` 说明原因。

---

## 4. log.md 写法

每行一条，格式：

```
- YYYY-MM-DD | 操作类型 | 文件路径 | 一句话说明 | 操作人
```

操作类型：`新增` / `更新` / `废弃` / `删除` / `修正`。

示例：

```
- 2026-06-25 | 新增 | system/architecture.md | 初始化架构全景 | 张三
```

---

## 5. 质量节奏建议

- **每次开发任务收尾**：回顾本次是否产生了应沉淀的知识。
- **每周**：扫一遍 `pitfalls/` 看是否有可关闭的坑。
- **每迭代/每月**：抽查 `system/` 与代码是否一致，清理 `deprecated` 文档。

---

## 6. 与代码仓库的关系

- 知识库**不修改任何源码**，只描述和记录。
- 当代码与知识库描述冲突时，**以代码为准**，并立即修正知识库。
- 知识库随项目一起进入版本控制（git），变更走正常评审。
