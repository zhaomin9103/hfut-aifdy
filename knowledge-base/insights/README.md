# 团队经验（Insights）

> 反复出现的模式、踩坑后总结出的最佳实践。**同一模式出现 ≥ 2 次** 才沉淀到这里，避免噪音。

---

## 怎么写一条经验

每条经验建议包含：

- **模式**：在什么场景下反复出现
- **做法**：推荐的标准做法
- **理由**：为什么这样更好
- **反例**：常见的错误做法（可选）
- **关联**：相关源码或文档

可以单独开文件（如 `pagination-pattern.md`），也可以先在本文件累积，量大后再拆分。

---

## 已沉淀的经验

### 1. 「页面 ↔ API 文件」一一对应

- **模式**：每新增一个后台管理功能，都对应一个独立页面和一组接口。
- **做法**：`views/Xxx.vue` 配 `api/xxx.js`，命名对应；接口集中导出。
- **理由**：定位快、职责清、改动隔离，AI 协作时也容易按名索骥。
- **关联**：`../modules/inventory.md`、`../conventions/coding-style.md`

### 2. 统一拆包，业务层只关心数据

- **模式**：所有接口返回 `{ code, message, data }`，到处都要剥一层。
- **做法**：在 `request.js` 响应拦截器统一拆包并统一弹错。
- **理由**：业务代码更干净，错误处理一处收口。
- **关联**：`../system/call-chains.md`、`src/api/request.js:28-53`

### 3. 配置驱动 Mock / 真实后端切换

- **模式**：演示用假数据、联调用真后端，频繁切换。
- **做法**：用 `VITE_USE_MOCK` 环境变量一键切，代码零改动。
- **理由**：演示与联调解耦，避免改代码引入风险。
- **关联**：`../system/service-dependencies.md`、`vite.config.js:8`
